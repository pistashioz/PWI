Fetching Data from APIs

When we talk about fetching data from APIs, we’re referring to the process of making HTTP requests to specific endpoints provided by a server.

The client initiates a request through HTTP, utilizing methods like GET, POST, DELETE, or PUT. These correspond to the CRUD operations:

GET Method

The GET method is used to retrieve information from the given server using a given URI. Requests using GET should only retrieve data and should have no other effect on the data.

Characteristics

  • Safe: They should only retrieve data and should not change the data on the server.
  • Idempotent: This means that making multiple identical requests has the same effect as making a single request.
  • Cacheable: GET requests can be cached and stored for later use, which is efficient for the network.
  • No Data Body: GET requests should not have a request body, so all the data is sent in the URI.

Typical Use Cases

  • Retrieving a webpage or other resources.
  • Querying a database for specific information.

Example

Imagine we have an API for a library system, and we want to fetch a list of books. Our GET request might look like this:

GET /api/books

And if we’re looking for a specific book by ID, we might use a request like:

GET /api/books/123

If we need to filter the books by a category, for instance, we could use a query parameter:

GET /api/books?category=Novel

When the server receives this request, it will process it and return a response. The response typically has a status code, headers, and a body. For example:

HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: ...

[
    {
        "id": 123,
        "title": "The Great Gatsby",
        "author": "F. Scott Fitzgerald"
    },
    ...
]
  • Content-Length header in HTTP requests

    The Content-Length header in HTTP requests is used to indicate the size of the body data that is being sent to the server. This header is not always necessary in every type of HTTP request. Here's when and why you might include it:

    • GET request: The Content-Length would be present in the response, not the request. It informs the client about the size of the response body.
    • POST and PUT Requests: These requests often include a body. The Content-Length header is important here because it tells the server how much data it needs to read from the incoming stream. This helps the server to properly receive and parse the request body.
    • DELETE Requests: Typically, DELETE requests do not contain a body as they are used to remove a resource identified by the URI. Therefore, the Content-Length header is generally not required or included.
    • PATCH Requests: Similar to POST and PUT, PATCH requests usually include a body containing the changes to be applied to the resource. The Content-Length header is useful for indicating the size of this body.
    • HEAD Requests: HEAD requests do not have a body; they are used to retrieve the headers for a specific resource. As such, the Content-Length header is not included in the request, although it may be present in the response to indicate the size of the body if the request had been a GET instead of HEAD.

    In practice, when making HTTP requests using tools like fetch or Axios in JavaScript, the Content-Length header is often set automatically based on the size of the request body provided. We typically don't need to set this header manually unless we have a specific reason to do so.

    Here's how our example requests would look if we were to include the Content-Length header:

    POST /api/books
Content-Type: application/json
Content-Length: 74

{
    "title": "1984",
    "author": "George Orwell",
    "year": 1949
}

PUT /api/users/123
Content-Type: application/json
Content-Length: 70

{
    "name": "Updated Name",
    "email": "updated@example.com"
}

DELETE /api/users/123
// No Content-Length header as no body is sent.

PATCH /api/users/123
Content-Type: application/json
Content-Length: 45

{
    "email": "newemail@example.com"
}

HEAD /api/users/123
// No Content-Length header as no body is sent.

The 200 OK status code indicates that the request was successful, and the JSON array is the body of the response, containing the list of books or the details of a single book.

In our Vue.js application, we could use this GET request to populate a list of books or display the details of a single book. Here's a simplified example using the Fetch API:

const fetchBooks = async () => {
  try {
    const response = await fetch('/api/books');
    if (!response.ok) {
      throw new Error('Network response was not ok');
    }
    const books = await response.json();
    console.log(books);
  } catch (error) {
    console.error('There has been a problem with your fetch operation:', error);
  }
};

fetchBooks();

In the fetchBooks function, we send a GET request to the server, and if successful, we print the list of books to the console.

POST Method

The POST method is used to submit data to be processed to a specified resource. Unlike GET requests, POST requests are not safe or idempotent, which means they can create changes on the server and the same request can have different results.

Characteristics

  • Not Safe: They are intended to change the server state.
  • Not Idempotent: Multiple identical requests may have different effects, such as submitting a form multiple times.
  • Data Body: POST requests typically include data in the request body rather than in the URI.

Typical Use Cases

  • Creating a new record in a database, such as a new user registration.
  • Submitting form data.

Example

Continuing with the library system example, let's say we want to add a new book to the library's catalog. Our POST request might include data about the book in the request body:

POST /api/books
Content-Type: application/json

{
    "title": "1984",
    "author": "George Orwell",
    "year": 1949
}

When the server receives this request, it will process the provided data and create a new book record. The server might respond with a status code of 201 Created to indicate that the book was successfully added to the catalog.

In Vue.js, using Axios or the Fetch API, a POST request could look like this:

const addBook = async (bookData) => {
  try {
    const response = await fetch('/api/books', {
      method: 'POST',
      headers: {
        'Content-Type': 'application/json',
      },
      body: JSON.stringify(bookData),
    });

    if (!response.ok) {
      throw new Error('Network response was not ok');
    }

    const result = await response.json();
    console.log(result);
  } catch (error) {
    console.error('There has been a problem with your fetch operation:', error);
  }
};

const newBook = {
    title: "1984",
    author: "George Orwell",
    year: 1949
};

addBook(newBook);

Here, addBook is an async function that sends a POST request to the server with the new book data. If the request is successful, it logs the result, which might include details of the created book or a confirmation message.

PUT Method

The PUT method is designed for updating existing resources or creating a new one if it doesn't exist at the specified URL. Unlike the POST method, the PUT method is idempotent, meaning that repeated identical requests should have the same effect as a single request.

Characteristics

  • Update or Create: Used to update existing resources or create new ones if they don't exist.
  • Idempotent: Making the same PUT request multiple times will result in the same state of the server (assuming no other changes between requests).
  • Data in Request Body: The data to update the resource is typically included in the request body, not in the URI.

Typical Use Cases

  • Modifying a record in a database.
  • Updating user profile information.

Example

Let's say we have a user in the system with an ID of 123, and we want to update their name and email. The PUT request would look like this:

PUT /api/users/123
Content-Type: application/json

{
    "name": "Updated Name",
    "email": "updated@example.com"
}

Upon receiving this request, the server will update the user with ID 123 with the provided name and email. If the user does not exist, it may create a new user with the provided details, depending on the API's implementation.

In Vue.js, using Axios or the Fetch API, we could send a PUT request like this:

const updateUser = async (userId, userData) => {
  try {
    const response = await fetch(`/api/users/${userId}`, {
      method: 'PUT',
      headers: {
        'Content-Type': 'application/json',
      },
      body: JSON.stringify(userData),
    });

    if (!response.ok) {
      throw new Error('Network response was not ok');
    }

    const result = await response.json();
    console.log(result);
  } catch (error) {
    console.error('There has been a problem with your fetch operation:', error);
  }
};

const updatedUser = {
    name: "Updated Name",
    email: "updated@example.com"
};

updateUser(123, updatedUser);

Here, updateUser is an async function that sends a PUT request to the server with the user's updated data. If the request is successful, it logs the result which would typically include the updated user information.

DELETE Method

The DELETE method is used to remove a specified resource from the server.

Characteristics

  • Purpose: Used to delete a resource identified by a URI.
  • Idempotent: Like PUT, the DELETE method is idempotent. This means if we DELETE a resource, it's removed. Repeating the DELETE request will end up having the same result – the resource is still gone.
  • Parameters: It typically includes an identifier in the URL to specify the resource that should be deleted.

Typical Use Cases

  • Removing a user from a database.
  • Deleting a file or entry.

Example

Suppose we want to remove a user with ID 123 from the system. The DELETE request would look like this:

DELETE /api/users/123

When the server processes this request, it will delete the user with the specified ID. After a successful operation, the server might respond with a status code of 200 OK or 204 No Content to indicate that the request was processed successfully but there's no content to send back.

In Vue.js, sending a DELETE request using Axios or the Fetch API could be done as follows:

const deleteUser = async (userId) => {
  try {
    const response = await fetch(`/api/users/${userId}`, {
      method: 'DELETE',
    });

    if (!response.ok) {
      throw new Error('Network response was not ok');
    }

    console.log('User deleted successfully');
  } catch (error) {
    console.error('There has been a problem with your fetch operation:', error);
  }
};

deleteUser(123);

In the deleteUser function, we send a DELETE request to the server specifying the user ID to be deleted. If the operation is successful, we log a confirmation message.

The DELETE method is straightforward but powerful. It's crucial to ensure that our API endpoints are secure and that only authorized users can perform delete operations to prevent unauthorized data deletion.

PATCH Method

The PATCH method is used to apply partial updates to a resource.

Characteristics

  • Partial Update: Unlike PUT, which typically requires sending the full entity, PATCH is used to send only the changes to the entity.
  • Idempotent: Technically, PATCH can be non-idempotent, but it is usually implemented in an idempotent manner.
  • Parameters: The parameters include the changes to be applied, often in the form of a diff, patch, or partial resource representation.

Typical Use Case

  • Updating a subset of fields for a resource. For instance, if we have a user profile with 10 fields and we only need to update the email address and profile picture, we would use PATCH.

Example

If we want to update just the email address of a user with ID 123, a PATCH request might look like this:

PATCH /api/users/123
Content-Type: application/json

{
    "email": "newemail@example.com"
}

In Vue.js, using Axios or the Fetch API, a PATCH request could look like this:

const updateUserEmail = async (userId, newEmail) => {
  try {
    const response = await fetch(`/api/users/${userId}`, {
      method: 'PATCH',
      headers: {
        'Content-Type': 'application/json',
      },
      body: JSON.stringify({ email: newEmail }),
    });

    if (!response.ok) {
      throw new Error('Network response was not ok');
    }

    console.log('User email updated successfully');
  } catch (error) {
    console.error('There has been a problem with your fetch operation:', error);
  }
};

updateUserEmail(123, "newemail@example.com");

The updateUserEmail function is an asynchronous function that takes a userId and newEmail as arguments. It sends a PATCH request to the server to update a specific field of a resource. The request body contains the new email address, which is the only field being updated. If the request is successful, it logs a confirmation message. Error handling is done using a catch block that logs any exceptions that occur during the fetch operation.

HEAD Method

The HEAD method is similar to GET, but it does not return the body in the response. It is used to retrieve the metadata of a specified resource, such as headers, without having to download the entire body.

Characteristics

  • No Body: The server will not send back the resource's body in the response.
  • Retrieve Headers: Useful for retrieving the headers to understand the resource's state, check for updates, or to see if the resource exists before downloading it.

Typical Use Cases

  • Checking if a resource exists before deciding to perform a GET request.
  • Checking if a resource has been modified since the last access.

Example

If we want to check the headers for a user resource with ID 123, a HEAD request might look like this:

HEAD /api/users/123

In Vue.js, we would handle this similarly to a GET request but without expecting a response body:

// Example using Fetch API to make a HEAD request in Vue.js

const checkResourceExists = async (resourceUrl) => {
  try {
    const response = await fetch(resourceUrl, { method: 'HEAD' });

    if (response.ok) {
      // The resource exists, and we can check headers for more information.
      console.log(`Content-Type: ${response.headers.get('Content-Type')}`);
      console.log(`Content-Length: ${response.headers.get('Content-Length')}`);
    } else {
      // The resource does not exist, or there was another error.
      console.error('Resource not found or other error');
    }
  } catch (error) {
    console.error('Network error:', error);
  }
};

// Call the function with the URL of the resource you want to check.
checkResourceExists('https://example.com/resource');

In this example, checkResourceExists function makes a HEAD request to the server for a given resource URL. If the resource exists, it logs the Content-Type and Content-Length from the headers. This information could be used to decide whether to make a full GET request to download the resource.

REST (Representational State Transfer)

REST is an architectural style that defines certain principles for developing network-based applications. The principles include:

  • Statelessness: The server does not keep any client state between requests. Each request from the client to the server must contain all the information the server needs to fulfill that request.
  • Client-Server Architecture: There’s a clear separation between the client (the frontend/user interface) and the server (backend/data storage), allowing each to evolve independently.
  • Uniform Interface: REST dictates that the API must have a uniform interface, which simplifies and decouples the architecture, which leads to each part being able to evolve independently.
  • Identification of Resources: Resources, anything that can be named, are identified using URIs (Uniform Resource Identifiers). For instance, /api/users could be an endpoint that provides access to user resources.
  • Representation of Resources: When a client fetches a resource, it gets a representation of that resource, typically in formats like JSON or XML.

HTTP Status Codes

The server's response to a client request comes with an HTTP status code, as shown in the picture below.

Some of the common HTTP status codes include:

  • 200 OK: The request was successful, and the server is returning the requested data.
  • 201 Created: A new resource has been created as a result of the POST request.
  • 404 Not Found: The server has not found anything matching the Request-URI.

HATEOAS (Hypermedia As The Engine Of Application State)

This principle allows clients to dynamically navigate the APIs by including hypermedia links with the responses. This way, the client doesn't need to hard-code URIs; it can instead discover them at runtime, making the API more discoverable and self-descriptive.

Security Considerations

RESTful APIs often use HTTPS (HTTP Secure) to encrypt the communication between the client and server, providing a secure way to exchange data. In addition, to control access to resources, authentication and authorization mechanisms are implemented, like OAuth2 for authentication or JWT (JSON Web Tokens) for both authentication and authorization.

Generic API Utility Functions

The creation of modular, reusable functions for making API requests is a fundamental aspect of modern web development. The purpose of these utility functions is twofold:

  1. Reusability: They allow us to write common API-related logic once and use it across different parts of the application. This not only saves time but also helps in maintaining consistency throughout our application’s codebase.
  1. Abstraction: These functions abstract the complexities involved in making HTTP requests, handling errors, and processing responses. This allows other parts of our application to use these functions without concerning themselves with the underlying details of the API communications.

Common Operations

  • CRUD Operations: Functions are typically implemented for each HTTP method (GET, POST, PUT/PATCH, DELETE) to cover the full spectrum of CRUD (Create, Read, Update, Delete) operations.
  • Headers: The functions include the ability to set common HTTP headers, such as Content-Type for specifying the media type of the resource or Authorization for securing the API requests.
    • Setting HTTP Headers in Vue.js

      In the context of making HTTP requests, headers are an essential part of the request and response lifecycle. They define the operating parameters of an HTTP transaction. In our Vue.js applications, when making API requests, we often need to set HTTP headers for various reasons, such as content negotiation or authentication.

      When we make an API request, we can set HTTP headers using the Fetch API or libraries like Axios. Here's how we do it:

      Using Fetch API

      The Fetch API allows us to make network requests similar to XMLHttpRequest (XHR). It is promise-based and provides a more powerful and flexible feature set. Here's an example of how we can set headers using Fetch:

      fetch('<https://api.example.com/data>', {
  method: 'GET', // or 'POST', 'PUT', 'DELETE', etc.
  headers: {
    'Content-Type': 'application/json',
    'Authorization': 'Bearer your-token-here',
    // Other headers can be set here as well
  }
})
.then(response => response.json())
.then(data => console.log(data))
.catch(error => console.error('Error:', error));

      In the above example:

      • The headers property in the fetch options is an object where the keys are the names of the headers, and the values are the values we want to set for those headers.
      • Content-Type is commonly used to specify the nature of the data in the request body, which in this case is JSON.
      • Authorization is often used to pass credentials, such as when using bearer tokens for OAuth.

      Using Axios

      Here's an example of setting headers with Axios:

      axios.get('<https://api.example.com/data>', {
  headers: {
    'Content-Type': 'application/json',
    'Authorization': 'Bearer your-token-here'
  }
})
.then(response => console.log(response.data))
.catch(error => console.error('Error:', error));

      Or for a POST request with a body:

      axios.post('<https://api.example.com/data>', {
  /* our data here */
}, {
  headers: {
    'Content-Type': 'application/json',
    'Authorization': 'Bearer your-token-here'
  }
})
.then(response => console.log(response.data))
.catch(error => console.error('Error:', error));

      In the Axios examples:

      • The second argument for axios.get and the third argument for axios.post is the configuration object where we can set headers.
      • Axios automatically sets the Content-Type header for requests with a body, but it can be overridden as shown.

      Commonly Used HTTP Headers

      • Content-Type: Indicates the media type of the resource.
      • Authorization: Carries credentials containing the authentication information of the client for the resource being requested.
      • Accept: Tells the server what content types the client can process.
      • Cache-Control: Directs the caching mechanisms in the request and response chain.
      • Custom-Headers: Sometimes, we might need to set custom headers specific to our API or web service.

      Setting the right HTTP headers is crucial for ensuring that our requests are handled correctly by the server and that responses are appropriately understood by our client application. In a Vue.js application, we might often set these headers in a service or utility layer that handles API interactions, ensuring that they're consistently applied across our app.

Error Handling

  • Consistent Error Handling: A uniform method of handling errors across all API requests helps in debugging and provides a predictable outcome for error scenarios.
  • HTTP Status Codes: These functions interpret HTTP status codes and translate them into meaningful error messages or responses, which can then be used to inform the user or trigger additional logic within the application.

Configuration

  • Base URL: Configuring a base URL parameter allows the functions to adapt to different API environments (like development or production) without changing the endpoint logic.
  • Timeouts: These utility functions can be configured to handle timeouts, ensuring that the application remains responsive even if an API request takes too long or fails to get a response.

Example Code Explanation

GET Utility Function

export async function get(apiBaseUrl, endpoint) {
  try {
    const response = await fetch(`${apiBaseUrl}/${endpoint}`);
    return handleResponse(response);
  } catch (error) {
    console.error(`Error fetching ${endpoint}:`, error);
    throw error;
  }
}

This get function is an asynchronous function that sends a GET request to a specified endpoint at the apiBaseUrl. It uses JavaScript’s fetch API to perform the request. If the request is successful, it calls handleResponse (another function that we assume is defined elsewhere) to process the response. If there’s an error (like a network issue), it logs and re-throws the error.

POST Utility Function

export async function post(apiBaseUrl, endpoint, data) {
  try {
    const response = await fetch(`${apiBaseUrl}/${endpoint}`, {
      method: "POST",
      headers: {
        "Content-Type": "application/json",
      },
      body: JSON.stringify(data),
    });
    return handleResponse(response);
  } catch (error) {
    console.error(`Error posting to ${endpoint}:`, error);
    throw error;
  }
}

Here, the post function is similar to the get function but is designed to send POST requests. It includes additional parameters for the request method, headers, and the request body. The body is the data to be sent which is stringified into JSON format.

Handling Responses

async function handleResponse(response) {
  if (!response.ok) {
    const errorMessage = await response.text();
    throw new Error(
      `API request failed with status ${response.status}: ${errorMessage}`
    );
  }
  const data = await response.json();
  return data;
}

The handleResponse function processes the response from the API request. If the response is not ok (the ok property is a shorthand for status codes 200-299), it gets the response text and throws an error with a message including the status code and the error message from the response. If the response is ok, it parses the JSON body of the response and returns the resulting data.

Using Pinia Actions in Components

Pinia is the state management library for Vue, and actions in Pinia are methods associated with the store that typically handle asynchronous operations like fetching data from an API.

Integrating API Utility Functions in Pinia Actions

We can integrate the API utility functions within Pinia actions to perform CRUD operations and update the store’s state accordingly.

import * as api from '../api/api'; // Importing all exported functions from api.js

const DOG_API_BASE_URL = 'https://dog.ceo/api';

async fetchDogBreeds() {
  try {
    // Make an API GET request using the imported 'get' method from api.js
    const data = await api.get(DOG_API_BASE_URL, 'breeds/list/all');
    // Assuming the response has a 'message' object with breeds as keys,
    // extract those keys and assign them to the 'dogBreeds' array.
    this.dogBreeds = Object.keys(data.message);
  } catch (error) {
    // If there's an error in the API request, log it to the console and rethrow it.
    console.error("Error in store fetching dog breeds:", error);
    throw error;
  }
},

In this example, fetchDogBreeds is a Pinia action that uses the get function from the API utilities to fetch the list of dog breeds from the DOG_API_BASE_URL. The action updates the dogBreeds state within the store with the list of breeds retrieved from the API response.

  • async fetchDogBreeds(): An asynchronous action method within the Pinia store.
  • api.get: Calls the imported get utility function, passing in the API base URL and the specific endpoint 'breeds/list/all'.
  • this.dogBreeds: Assumes there is a state property dogBreeds in the Pinia store that will be updated with the results.
  • Object.keys()

    In JavaScript, Object.keys() is a method that returns an array of a given object's own property names, in the same order as we get with a normal loop. The usage of Object.keys() in the example above is a way to extract keys from the message object, which presumably contains the breeds as keys.

    Let's say the API response for data.message looks like this:

    {
  "message": {
    "bulldog": ["boston", "english", "french"],
    "terrier": ["american", "australian", "bedlington"],
    // ... other breeds
  }
}

    When we call Object.keys(data.message), it will return an array of the breed names, ignoring the sub-breeds:

    javascriptCopy code
["bulldog", "terrier", /* ... other breeds */]

    So, in our component's dogBreeds array, we'll have a list of the main dog breeds, which we can then use to render a list in your Vue.js application or perform other actions. The Object.keys() method is very useful for transforming objects into arrays when we're interested in the keys (or property names).

    → If we want to access the sub-breeds along with the main breeds, we would need to iterate over the values of each key in the data.message object. We can do this by using Object.keys() to get the main breeds and then map over those keys to return both the breed and its sub-breeds. Here's how we could modify the fetchDogBreeds function to also include the sub-breeds:

    async fetchDogBreeds() {
  try {
    const data = await api.get(DOG_API_BASE_URL, 'breeds/list/all');
    // Create an array of breeds, where each breed is an object with name and sub-breeds
    this.dogBreeds = Object.keys(data.message).map(breed => {
      return {
        breed,
        subBreeds: data.message[breed]
      };
    });
  } catch (error) {
    console.error("Error in store fetching dog breeds:", error);
    throw error;
  }
},

    With this approach, this.dogBreeds will be an array of objects, and each object will have a breed property for the main breed's name and a subBreeds property for the array of sub-breeds. Here's an example of what the dogBreeds array will look like:

    [
  {
    breed: "bulldog",
    subBreeds: ["boston", "english", "french"]
  },
  {
    breed: "terrier",
    subBreeds: ["american", "australian", "bedlington"]
  },
  // ... other breeds
]

    This structure gives us a detailed hierarchy of dog breeds, allowing us to access both the main breed and its sub-breeds for use in our application.

    Rendering Dog Breeds and Sub-breeds in Vue.js with Pinia Store

    Step 1: Create a Pinia Store

    First, set up a Pinia store to manage the dog breeds data. This store will handle fetching and storing the breeds and their sub-breeds.

    // store/dogs.js
import { defineStore } from 'pinia';

export const useDogStore = defineStore('dogStore', {
  state: () => ({
    breeds: {}
  }),
  actions: {
    async fetchDogBreeds() {
      try {
        const response = await fetch('<https://dog.ceo/api/breeds/list/all>');
        const data = await response.json();
        this.breeds = data.message; // Structure: { breed: [subbreeds] }
      } catch (error) {
        console.error("Error fetching dog breeds:", error);
      }
    }
  }

Error Handling in Components

When we call Pinia actions within our Vue components, it is essential to handle potential errors that may occur during the asynchronous operation. We can use a try-catch block to manage errors gracefully. 

created() {
  try {
    this.dogStore.fetchDogBreeds();
  } catch (error) {
    alert(error.message);
  }
},

In the above code:

  • created(): A Vue lifecycle hook that runs when the component is created.
  • this.dogStore.fetchDogBreeds(): Calls the fetchDogBreeds action from the dogStore Pinia store.
  • try-catch: Catches any errors that occur during the execution of the fetchDogBreeds action.
  • alert(error.message): Displays an alert with the error message if an error occurs.

This pattern ensures that our application can handle API request failures and prevent unhandled exceptions, which could lead to a poor user experience.

Important Concepts

  1. HTTP Methods and CRUD Operations
    • GET: Safe and idempotent, for retrieving data.
    • POST: For creating new resources, not safe or idempotent.
    • PUT: For updating entire resources, idempotent.
    • DELETE: For removing resources, idempotent.
    • PATCH: For applying partial updates, usually idempotent.
    • HEAD: For retrieving headers, similar to GET but without the body.
  1. REST Principles
    • Statelessness: No client state is stored on the server.
    • Client-Server Architecture: Separation of concerns.
    • Uniform Interface: Standardized way to interact with resources.
    • Resource Identification: Using URIs.
    • Representation: Resources represented in standard formats like JSON, XML.
  1. HTTP Status Codes
    • Common codes like 200 (OK), 201 (Created), 404 (Not Found).
  1. HATEOAS: Dynamic navigation through APIs using hypermedia links.
  1. Security: Using HTTPS, authentication, and authorization mechanisms.
