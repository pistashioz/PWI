Vue Router

Traditional Page Lifecycle

  • Initial Page Request: When we visit a traditional website, the browser sends a request to the web server, which then responds with an HTML page.
  • New Page Request: Each time we navigate to a new page (e.g., by clicking a link), the browser sends a new request to the server for that page. The server sends back the new HTML page, and the browser reloads the entire page, which includes re-downloading CSS, JavaScript, and other assets. This process sends a physical page view to analytics platforms like Google Analytics (GA) each time.

SPA Page Lifecycle

  • Initial Request: As with a traditional application, the initial visit to an SPA involves the browser requesting the page from the server, which sends back HTML, along with JavaScript and CSS required to run the application.
  • AJAX Request: Unlike traditional applications, navigation in a SPA doesn’t involve reloading the entire page. Instead, the browser updates the content dynamically through AJAX (Asynchronous JavaScript and XML) requests. These requests fetch data (usually in JSON format) without a full page reload. JavaScript then re-renders the UI with the new data. This process does not send physical page views to analytics platforms with each navigation, necessitating a different approach to tracking user interactions.

Key Points of SPAs

  • SPAs provide a more fluid and app-like user experience by reducing the number of full-page reloads.
  • SPAs are generally faster after the initial load since only data, not the whole page, is fetched during navigation.
  • SPAs can be more complex to develop and might require additional tolls for SEO and analytics tracking due to the lack of full-page reloads.

An Introduction

When we begin working with Vue.js to build a single-page application (SPA), one of the first steps is to set up the Vue Router, which is the official router for Vue.js. The Vue Router allows us to create navigable components within our application, enabling the user to move between different “views” or “pages” without the need for a full page reload.

1. Project Creation

  • Start by creating a new Vue.js project, often using the Vue CLI (Command Line Interface). During the setup process, we’ll have the option to include Vue Router in the project. Select this option to have Vue Router integrated from the start.

2. Directory Structure

  • Within the src directory of the Vue.js project, we will typically find several important sub-directories and files:
    • assets: This directory is meant for static assets like images, stylesheets, and global scripts that may be used across the app.
    • components: Components are reusable Vue instances that encapsulate template, logic, and styling for a part of the UI.
    • router: Here lies the heart of navigation for the SPA. The index.js file within the router directory defines all the routes for the application, mapping URLs to our components.
    • views: This directory holds the different “pages” or “views” of the app. In a Vue Router setup, each view is associated with a route.
    • App.vue: The main component that acts as the entry point for our application. It’s where we define the <router-view> component that will dynamically display the content based on the current route.
    • main.js: The main JavaScript file where we create the Vue instance, import and use the Vue Router, and mount the Vue app to the DOM.

3. Vue Router Setup

  • The Vue Router setup is initialized in the main.js file where we’ll import the router from the router directory. By passing the router instance to the Vue application, we allow Vue to handle the routing within the app, enabling navigation between different views defined in the views directory without reloading the page.

4. Routing in Action

  • Once the Vue Router is set up, navigating to a URL within the app will not request a new HTML page from the server. Instead, Vue Router will match the URL to a route defined in router/index.js and render the corresponding view within the <router-view> component of the App.vue.
// Imports the necessary functions from the vue-router package
import { createRouter, createWebHistory } from 'vue-router'
// Imports the HomeView component from the views directory
import HomeView from '../views/HomeView.vue'

// Creates a new router instance
const router = createRouter({
  // Uses the createWebHistory function to enable HTML5 history mode
  history: createWebHistory(import.meta.env.BASE_URL),
  // Defines the routes for the application
  routes: [
    {
      // The path for the home page
      path: '/',
      // The name for the route, which can be used for navigation
      name: 'home',
      // The component that will be rendered when the user navigates to this path
      component: HomeView
    },
    {
      // The path for the about page
      path: '/about',
      // The name for the route
      name: 'about',
      // This route level code-splitting
      // which generates a separate chunk (About.[hash].js) for this route
      // which is lazy-loaded when the route is visited.
      component: () => import('../views/AboutView.vue')
    }
  ]
})

// Exports the router instance for use in the application
export default router
  • Explanation of Concepts
    • Module Imports: The createRouter and createWebHistory functions are imported from the ‘vue-router’ package, which is essential for creating and configuring the router.
    • Component Imports: The HomeView component is imported, which will be used as a view for one of the routes.
    • Router Creation: createRouter is called to create a new router instance, which is essential for enabling routing in the Vue application.
    • History Mode: createWebHistory is used to create a history object that uses the browser’s history stack. This allows for pretty URLs that don’t include hashes(#).
    • Routes Definition: An array of route objects is defined, where each route is mapped to a component. In the example above, there are two routes:
      • The root path(’/’) is mapped to HomeView, which will be displayed when the user navigates to the home page.
      • The /about path is mapped to a lazy-loaded AboutView, which means the component for this route will only be loaded when the user navigates to the /about path, optimizing the initial load time.
    • Lazy Loading: The AboutView component is not imported at the top with the other imports. Instead, it’s imported inside a function that will only be executed when the /about path is visited. This is an example of code-splitting and lazy loading, which can improve the performance of a Vue application by reducing the size of the initial JavaScript bundle that needs to be loaded when the app starts.
    • Exporting the Router: Finally, the router instance is exported, which allows it to be imported and used in other parts of the application, like the main app initialization file.

main.js Setup

This section explains how to integrate the Vue Router into the main entry file of a Vue.js application, typically main.js.

// Importing the required modules from Vue and the App component
import { createApp } from 'vue'
import App from './App.vue'
// Importing the router configuration from the router directory
import router from './router'

// Importing the main stylesheet
import './assets/main.css'

// Creating a Vue application instance with the App component
const app = createApp(App)

// Associating the Vue Router with the Vue application instance
app.use(router)

// Mounting the Vue application to the DOM element with the id of 'app'
app.mount('#app')
  • Explanation of Concepts
    • Module Imports: The createApp function from Vue is imported along with the App component and the router setup.
    • Router Setup Import: The router configuration that was exported from the router/index.js file is imported into the main application file.
    • Application Instance: The createApp function is invoked with the App component to create a new Vue application instance.
    • Router Association: The app.use(router) call associates the imported router configuration with the application, enabling Vue Router in the app.
    • Mounting: Finally, app.mount(’#app’) mounts the application instance to the DOM, activating Vue’s reactivity and rendering system.

App.vue Setup

This section focuses on how to use Vue Router within the App.vue component to define navigation links and display the current route’s component.

<template>
  <header>
    <!-- Static assets like images can be included directly in templates -->
    <img alt="Vue logo" class="logo" src="@/assets/logo.png">
    <div class="wrapper">
      <!-- The HelloWorld component with a message prop -->
      <HelloWorld msg="You did it!" />
    </div>
    <nav>
      <!-- Using RouterLink components to create navigational links -->
      <RouterLink to="/">Home</RouterLink>
      <RouterLink to="/about">About</RouterLink>
    </nav>
  </header>
  <!-- The RouterView component where the matched route's component will render -->
  <RouterView />
</template>

<script>
// Importing RouterLink and RouterView components from vue-router
import { RouterLink, RouterView } from 'vue-router'

export default {
  name: 'App',
  components: {
    RouterLink,
    RouterView
  }
}
</script>
  • Explanation of Concepts
    • RouterLink: The <RouterLink> component is used to define navigational links in the application. It accepts a to prop, which specifies the path to navigate to when the link is clicked. These replace the traditional <a> tags for in-app navigation.
    • RouterView: The <RouterView> component is a placeholder for rendering the component that corresponds to the current route. When the route changes, <RouterView> automatically updates to display the correct component.
    • Component Import and Registration: Inside the <script> section, RouterLink and RouterView are imported from the ‘vue-router’ package and registered in the components object, making them available within the App.vue template.

Dynamic Routing

A powerful feature in Vue Router that allows us to create routes that can match complex patterns and dynamically change based on the URL.

What is Dynamic Routing?

  1. Pattern Matching: Dynamic routing enables us to define patterns for routes that can match multiple URL structures. For example, we might have a pattern like /users/:id where :id is a dynamic segment that can represent any user’s ID.
  1. Component Re-usability: Instead of creating a new route and component for each user, dynamic routing lets us map a single route pattern to a component. This component then renders the appropriate content by accessing the dynamic segment of the URL.
  1. Example Usage: Consider a User component that displays user profiles. With dynamic routing, we only need to create on User component. Vue Router will render this component for all user profile routes, such as /users/john, /users/jane, etc. The dynamic part of the route (:id) will change, but the same User component will be used to render the page each time.
  1. Dynamic Segment: The dynamic part of the URL (the user ID in this case) is accessible within the component via this.$route.params. This allows the component to fetch and display data that is specific to each user.
  1. Flexibility: Dynamic routing provides the flexibility to build a SPA with a clean and organized URL structure. It greatly simplifies the route configuration when dealing with user-generated or variable content.
  1. Use Cases: It’s particularly useful for situations like user profiles, product pages, or any scenario where the content changes based on an identifier in the URL.

Basic Dynamic Routing

const User = {
  template: '<div>User</div>',
}

const routes = [
  { path: '/users/:id', component: User },
]
  • User Component: This is a simple Vue component with a placeholder template.
  • Dynamic Segment: The route /users/:id contains a dynamic segment :id. This means any path matching /users/x will render the User component, where x represents a dynamic user ID.

Dynamic Routing with Parameters

Dynamic routing allows URLs like /user/foo and /user/bar to both map to the User component.

const User = {
  template: '<div>User {{ $route.params.id }}</div>',
}
  • Dynamic Segment Usage: The id from the URL is accessed within the component using $route.params.id, enabling the component to display or use the dynamic part of the URL.
  • Parameter Exposure: Vue Router exposes dynamic segments as properties on $route.params.

Multiple Dynamic Segments

Vue Router supports multiple dynamic segments in the same route.

PatternMatched Path$route.params
/users/:username/users/eduardo{ username: 'eduardo' }
/users/:username/posts/:postId/users/eduardo/posts/123{ username: 'eduardo', postId: '123' }
  • Multiple Segments: Routes can contain multiple dynamic segments, such as /users/:username/posts/:postId, which maps to corresponding fields on $route.params.
Understanding $route.params
  1. Global Availability: When we use Vue Router, it injects the $route object into every Vue component. This object contains information about the current route, such as the matched path, query parameters, and the named parameters (dynamic segments).
  1. Dynamic Segments in Routes: When we define a route with a path like /users/:username, the :username is a placeholder for whatever value is in that segment of the URL when the route is visited.
  1. Accessing Parameters in Components:
    • Inside any Vue component, we can directly access $route.params in our JavaScript with this.$route.params and in our templates with {{ $route.params.someParam }}.
    • The dynamic segments of the URL are available on this.$route.params as an object.
    • For the route /users/:username, if you navigate to /users/eduardo, Vue Router will set $route.params to { username: ‘eduardo’ }.
  1. Multiple Parameters: If we have a route with multiple parameters, like /users/:username/posts/:postId, navigating to /users/eduardo/posts/123 will set $route.params to { username: 'eduardo', postId: '123' }.
  1. Using Params in Templates: In our template, we can display a route parameter by using mustache syntax {{ }}. For example:
    <div>User: {{ $route.params.username }}</div>
  1. Reactivity:
    • The $route object is reactive. If the URL changes and the active route still matches the same component (e.g., going from /users/john to /users/jane), the component will update reactively to reflect the new params.
    • However, since the component instance itself is reused, lifecycle hooks are not called again. To respond to params changes, we can use a watcher on $route or the beforeRouteUpdate hook.
  1. Location in Code:
    • We don’t explicitly define $route.params in our scripts. Vue Router takes care of populating it based on the current URL and the route definitions that have dynamic segments.
    • In a component, we usually access it in computed properties, methods, or directly in the template to render dynamic content based on the route.
  1. Part of Vue Router's Context: When a component is rendered by Vue Router, it's provided with a route context that includes the $route object. This context is what allows the component to access route-specific data, including dynamic route parameters.

Practical Example

Imagine a profile page for users in a social media app. The route may look like /users/:id, where id is the dynamic segment representing the user ID. When a visitor navigates to /users/42, the $route.params object inside our UserProfile component will be { id: '42' }. Our component can then use this ID to fetch and display the correct user profile from the server.

By using $route.params, Vue Router provides a way to create dynamic and user-friendly URLs for our application, making it possible to create a single component that can handle multiple routes and display the appropriate content based on the URL.

Reacting to Params Changes

Handling changes in route parameters when the same component is used for multiple routes:

  • Component Reuse: When navigating from /user/foo to /user/bar, Vue Router reuses the User component instance.
  • Lifecycle Hooks: The component’s lifecycle hooks won’t be called again since it’s the same instance.
  • Reacting to Changes: To respond to param changes without re-creating the component use $watch or the beforeRouteUpdate guard.

Advanced Dynamic Routing Patterns

Vue Router’s routing can handle more complex patterns

  • Type Params: Dynamic segments can be further specified to match only certain patterns such as numbers or strings using regex.
  • Repeatable Params: Segments can be made repeatable by adding a + or * suffix, matching multiple segments in the URL.
  • Optional Params: Dynamic segments can be made optional with a ? suffix, allowing for more flexible URL structures.
const routes = [
  // Matches only numbers for orderId
  { path: '/order/:orderId(\\d+)' },
  // Matches anything for productName
  { path: '/product/:productName' },
  // Matches repeated segments for chapters
  { path: '/chapters/:chapters+' },
  // Matches optional userId
  { path: '/users/:userId?' },
]

Nested Routes

Nested routes in Vue Router allow us to reflect the hierarchical structure of the user interface within the URL path. This is a common pattern in many web applications where the user interface is composed of components nested at multiple levels.

What are Nested Routes?

  1. Hierarchical Interface: In a real application, the user interface (UI) often has multiple layers of components. For example, a main layout component might contain a sidebar component, which in turn might contain a navigation list component, and so on.
  1. URL Segments Corresponding to UI Structure: Nested routes enable the URL to mirror this hierarchical structure. For instance, the URL /dashboard/profile/edit might represent a structure where Dashboard is the parent component, which has a Profile component nested within it, and within Profile, there is an Edit component.
  1. Router Configuration: In Vue Router, nested routes are defined by giving a route a children array in the routes configuration.
const routes = [
  {
    path: '/dashboard',
    component: Dashboard,
    children: [
      {
        path: 'profile',
        component: Profile,
        children: [
          {
            path: 'edit',
            component: EditProfile
          }
        ]
      }
    ]
  }
];

In this structure:

  • The Dashboard route is the parent route
  • The Profile route is nested within Dashboard and would correspond to /dashboard/profile in the URL.
  • The EditProfile route is nested within Profile and would correspond to /dashboard/profile/edit in the URL.

Example of Nested Routes

const User = {
  template: '<div class="user"><router-view/></div>',
}

const UserHome = { template: '<div>Home</div>' }
const UserProfile = { template: '<div>Profile</div>' }
const UserPosts = { template: '<div>Posts</div>' }

const router = new VueRouter({
  routes: [
    {
      path: '/user/:id',
      component: User,
      children: [
        // UserHome will be rendered inside User's <router-view>
        // when /user/:id is matched
        { path: '', component: UserHome },
        
        // UserProfile will be rendered inside User's <router-view>
        // when /user/:id/profile is matched
        { path: 'profile', component: UserProfile },
        
        // UserPosts will be rendered inside User's <router-view>
        // when /user/:id/posts is matched
        { path: 'posts', component: UserPosts },
      ]
    }
  ]
})
  • Route Definitions: Routes are defined with a path and a component. The children array within a route definition is used for nesting.
  • Nested <router-view>: The User component includes a <router-view/>, which acts as a placeholder for child components. This is where UserHome, UserProfile, or UserPosts will be rendered, depending on the current URL.
  • Empty Child Path: An empty path in a child route ({ path: ‘’, component: UserHome }) means that UserHome will be the default component displayed when the parent route (/user/:id) is matched.
  • Nested Paths: Child routes append their path to be parent’s path. So, UserProfile corresponds to /user/:id/profile and UserPosts corresponds to /user/:id/posts.
  • Dynamic Segment: :id is a dynamic segment, meaning it will match any value, and the value can be accessed in the components as $route.params.id.

This setup allows you to build a UI where navigating to /user/foo, /user/foo/profile, and /user/foo/posts would render the User component with UserHome, UserProfile, and UserPosts respectively, within it.

Programmatic Navigation

A powerful feature of Vue Router that allows you to navigate users around an application without the need for <a> tags or any sort of link element. Instead, we can use Vue Router’s instance methods to change routes programmatically.

  1. RouterLink for Declarative Navigation
    1. Normally, Vue Router uses the <RouterLink> component to declaratively navigate to different routes. This component is used to generate <a> tags in the rendered HTML, which when clicked navigate to the specified route.
    1. For example, <RouterLink to="/user/1234">Go to user #1234</RouterLink> would render as <a href="/user/1234">Go to user #1234</a>.
  1. Custom Tag with RouterLink
    1. Sometimes, we might need a different HTML element to act as a navigation trigger, not just an anchor tag. Vue Router allows us to specify the tag we want to render by using the tag prop on <RouterLink>.
    1. For instance, <RouterLink to="/user/1234" tag="li">Go to user #1234</RouterLink> would result in <li>Go to user #1234</li> being rendered.

Programmatic Navigation Methods

router.push

  • router.push(location, onComplete?, onAbort?) is used to navigate to a different URL and create a new entry in the history stack. This allows users to click the browser’s back button to return to the previous URL.
// Navigating to a user's profile page
router.push('/user/1234')
// The URL will change to '/user/1234', and the history stack is updated.

router.replace

  • router.replace(location, onComplete?, onAbort?) behaves like router.push but does not leave a history record. The current navigation entry is replaced with the new one. This means the user cannot navigate to the previous route by clicking the browser back button.
// Replacing the current route with a new one
router.replace('/user/1234')
// The URL will change to '/user/1234', but no new history entry is created.

router.go

  • router.go is similar to window.history.go(n), allowing you to navigate through the history stack by a relative number. A positive number moves forward, and a negative number moves backward.
// Going back one step in the history stack
router.go(-1)
// Equivalent to pressing the browser's back button.

// Going forward two steps in the history stack
router.go(2)
// Equivalent to pressing the browser's forward button twice.

Practical Application

Imagine an application where a user interacts with a form, and upon submission, we want to navigate to a confirmation page:

// After form submission
submitForm() {
  // ... form handling logic ...

  // Then navigate to the confirmation page
  this.$router.push('/confirmation');
}

In this scenario, router.push is used inside a method to change the route when a form is submitted programmatically. This is just one of many scenarios where programmatic navigation enhances the user experience by providing seamless transitions without manual URL changes or link clicks.

Named Routes

Named routes in Vue Router offer a convenient way to reference routes in our application without using the full path URL. This can be particularly useful for improving code readability and making it easier to update paths throughout our application since we only need to update the route configuration rather than every link.

What are Named Routes?

  1. Identification by Name: Named routes allow us to assign a unique name to a particular route. Instead of using the route’s path to navigate or link to it, we can use its name.
  1. Convenience: When we need to perform navigations or create links to a route using a name can be simpler than specifying the entire path, especially if the path is complex or subject to change.
  1. Usage in <RouterLink>: We can use named routes in a <RouterLink> component by passing an object to the to prop with the name of the route and any parameters it requires.
  1. Usage in Programmatic Navigation: The same object used in <RouterLink> can be passed to router.push() when navigating programmatically.

Example of Named Routes

// Defining the route with a name
const router = new VueRouter({
  routes: [
    { path: '/user/:userId', name: 'user', component: User }
  ]
})

// Using the named route in a <RouterLink>
<RouterLink :to="{ name: 'user', params: { userId: 123 }}">Go to user #123</RouterLink>

// Using the named route in programmatic navigation
router.push({ name: 'user', params: { userId: 123 } })
  • Route Definition: In the route definition, we’re giving the route a name property of ‘user'. This name can then be used to refer to the route that matches the path /user/:userId.
  • In Templates: The <RouterLink> is using an object with the name of the route and the params that should be passed to that route. This tells Vue Router to navigate to the route named ‘user’ with a userId of 123.
  • In Scripts: The router.push() method uses the same object format as <RouterLink> to programmatically navigate to the named route with the specified parameters.

Named Views

Named views in Vue Router allow us to render multiple view components for the same route in different named outlets. This feature is particularly useful for creating complex layouts with sections that need to display different content simultaneously.

What are Named Views?

  1. Multiple Outlets: Instead of a single <router-view> outlet, we can have multiple outlets in our layout and assign different components to be rendered in each outlet by giving each <router-view> a name.
  1. Named Outlets: When defining our routes, we can specify which component should be rendered in each named outlet by using the components option, which is an object where the keys correspond to the names of the outlets.

How to Define Named Views

Here’s how we might define named views in our route configuration:

// Components to be rendered in the named views
const Foo = { template: '<div>foo</div>' }
const Bar = { template: '<div>bar</div>' }
const Baz = { template: '<div>baz</div>' }

// Router configuration with named views
const router = new VueRouter({
  routes: [
    {
      path: '/other',
      components: {
        default: Foo, // Component for unnamed or default outlet
        a: Bar,      // Component for the outlet named "a"
        b: Baz       // Component for the outlet named "b"
      }
    }
  ]
})

Usage in Templates

In our template, we would have multiple <router-view> elements, each with a name attribute corresponding to the keys in the components object of the route configuration:

<div id="app">
  <!-- Default unnamed router-view -->
  <router-view class="view one"></router-view>
  <!-- Named router-view "a" -->
  <router-view class="view two" name="a"></router-view>
  <!-- Named router-view "b" -->
  <router-view class="view three" name="b"></router-view>
</div>

Named Views with Nested Views

For more complex layouts, we can use named views within nested routes. This allows us to have nested components that also have named sections that change depending on the route. When doing this, we’ll also need to name nested <router-view> components.

Here’s a conceptual example of how nested named views might look:

  • Settings Page (/settings)
    • Emails Section (/settings/emails)
    • Profile Section (/settings/profile)

The settings route might have nested routes for emails and profile, each with their own named views for different subsections of the page.

Benefits of Named Views

  • Flexibility: Named views provide the flexibility to update different sections of a page in response to route changes.
  • Modularity: They help keep the UI modular by allowing us to map different components to different parts of the page layout, based on the current route.
  • Maintainability: Using named views can make our application easier to maintain, as we have a clear structure for where each piece of content should be rendered based on the route configuration.

Redirect and Alias

In Vue Router, “Redirect” and “Alias” are two different ways to handle navigation in your application. They both serve to guide users and maintain the integrity of the user interface without causing confusion or errors due to URL changes.

Redirect

A redirect is used when we want to send users from one path to another automatically. This is useful when we have changed the URL structure of our application or when we want to ensure that users and search engines find the new pages instead of outdated ones.

Example of Redirect

Suppose we had a page accessible at /a but decided to change it to /b. To ensure that users attempting to access /a are automatically taken to /b, we would set up a redirect like this:

const router = new VueRouter({
  routes: [
    { path: '/a', redirect: '/b' }
  ]
})

When a user visits /a, Vue Router automatically redirects them to /b, and the URL in the browser changes to reflect this.

Alias

An alias is used when we want a route to be accessible by more than one path. The alias allows the URL to remain as entered by the user, but it will be matched internally as if the user visited the original path. This means that the component associated with the original path will be rendered without changing the URL.

Example of Alias

If we want /a to serve the same component as /b without redirecting, we would set up an alias like this:

const router = new VueRouter({
  routes: [
    { path: '/a', component: A, alias: '/b' }
  ]
})

When a user visits /b, they will see the same component as if they visited /a, but the URL in the browser will remain /b.

Usage in Vue Router

Redirects and aliases provide flexibility in how we structure the navigation of our application:

  • User redirects when we want to guide users away from old URLs or when we want to consolidate multiple paths to a single target location.
  • Use aliases when we have multiple URLs that represent the same content or when we want to provide shortcuts or more readable URLs for specific paths without changing the underlying route configuration.

Wild Paths

Wildcard paths are utilized in Vue Router to match any route not previously matched by defined routes, commonly used to display a ‘404 Not Found’ page.

Syntax

The wildcard character * is used within the path string to denote a wildcard path. Vue Router supports custom regex patterns to fine-tune the matching criteria.

const routes = [
  // Matches any path and captures it under `$route.params.pathMatch`
  { path: '/:pathMatch(.*)*', name: 'NotFound', component: NotFound },
  // Matches anything after `/user-` and captures under `$route.params.afterUser`
  { path: '/user-:afterUser(.*)', component: UserGeneric },
];
  • The pathMatch parameter with a custom regex pattern captures all segments of unmatched paths.
  • Named routes like ‘NotFound’ can be used to navigate programmaticallly to catch-all route.
  • Wildcard routes should be placed last in the routes array to act as a default case.

Advanced Patterns

  • Vue Router’s syntax is inspired by express, allowing for optional params, zero or more/one or more requirements, and custom regex patterns.
  • It’s crucial to properly configure the server for Hisotry mode when using wildcard paths to ensure correct handling of deep linking.

Note: Always check the official Vue Router documentation for the latest updates and best practices related to wildcard paths and other advanced routing features.

Navigation Guards

Navigation guards in Vue Router are a powerful feature that allows us to control navigation flow within our Vue application. They are particularly useful for actins such as checking user authentication, logging navigation, and even canceling the navigation under certain conditions.

Types of Navigation Guards

Global Guards

  1. router.beforeEach

    This guard is called before every route change. It’s useful for actions that should happen before any route is resolved, like authentication checks.

    router.beforeEach((to, from, next) => {
  if (to.matched.some(record => record.meta.requiresAuth) && !authService.loggedIn()) {
    next({ path: '/login', query: { redirect: to.fullPath } });
  } else {
    next(); // always call next()!
  }
});

    This guard checks if the target route requires authentication (via meta.requiresAuth). If the user is not logged in (checked via authService.loggedIn()), it redirects them to the login page, preserving the intended destination (to.fullPath) so the user can be redirected back after successful login. If the user is already logged in, next() is called without any arguments, which means the navigation is allowed to proceed.

  1. router.afterEach

    This guard is executed after every navigation is confirmed. Unlike beforeEach, afterEach does not receive a next function and cannot affect the navigation. It’s typically used for tasks that do not require redirecting or canceling the navigation, like updating analytics or setting page titles.

    router.afterEach((to, from) => {
  // Update page title to match the route's metadata title
  document.title = to.meta.title || 'Default Page Title';
});

    In this example, after each navigation, the page title is updated to reflect the title specified in the destination route's meta field. If no title is set, it defaults to 'Default Page Title'. This guard is ideal for performing actions after the navigation has occurred, such as logging, changing classes on the body element, or other DOM manipulations.

Per-Route Guards

  1. beforeEnter

    This guard is defined on individual routes. It can be used to guard access to specific routes rather than globally

    const router = new VueRouter({
  routes: [
    {
      path: '/secure',
      component: SecureComponent,
      beforeEnter: (to, from, next) => {
        if (!authService.userHasAccess(to.name)) {
          next('/login');
        } else {
          next();
        }
      }
    }
  ]
});

    This guard is specific to the route for the path /secure. Before entering the route, it checks if the user has access to the route named 'secure' using authService.userHasAccess(to.name). If the user does not have access, they are redirected to the login page; otherwise, the navigation is allowed to proceed with next().

In-Component Guards

  1. beforeRouteEnter

    This guard is used to check conditions before the component is even loaded.

    export default {
  beforeRouteEnter(to, from, next) {
    if (!DataService.canEnter(to.params.id)) {
      next(false); // cancel the navigation
    } else {
      next();
    }
  }
};

    This guard is used in a component to prevent access before the component is created. It calls a service DataService.canEnter to check if the user can enter based on the id in the route parameters. If access is denied, next(false) is called to cancel the navigation; otherwise, next() allows it to proceed.

  1. beforeRouteUpdate

    This guard is called when the route changes, but the component is reused. It’s useful for reacting to changes in params or query without re-rendering the component.

    export default {
  beforeRouteUpdate(to, from, next) {
    this.loadData(to.params.id);
    next();
  }
};

    This guard is used when the same component is reused for a different route, but with changed parameters. It responds to the route changes by calling a method loadData to fetch new data based on the new id in the route parameters. next() is then called to finalize the navigation.

  1. beforeRouteLeave

    This guard is used to prevent a user from accidentally navigating away from the current view, like when unsaved changes exist.

    export default {
  data() {
    return {
      unsavedChanges: false
    };
  },
  beforeRouteLeave(to, from, next) {
    if (this.unsavedChanges) {
      const answer = window.confirm('You have unsaved changes. Are you sure you want to leave?');
      if (!answer) {
        next(false);
      } else {
        next();
      }
    } else {
      next();
    }
  }
};

    This guard is used to prevent users from leaving a component if there are unsaved changes (unsavedChanges). It prompts the user with a confirmation dialog when they try to navigate away. If the user confirms they want to leave, next() is called to proceed with the navigation. If the user cancels, next(false) is invoked to stop the navigation process.

How They Work

  • Guards receive to, from, and next arguments, allowing us to control the behavior based on the current and target route.
  • next is a function that must be called to resolve the guard, with the ability to redirect or cancel the navigation.

Common Uses

  • Authentication Checks: To ensure that users can only access routes they are authorized to view.
  • Data Fetching: To retrieve data before navigating to the next route.
  • Conditional Navigation: To prevent navigation based on certain criteria, such as unsaved changes.

Navigation Resolution Flow

  1. Navigation is triggered.
  1. Call leave guards in deactivated components.
  1. Call global beforeEach guards.
  1. Call beforeRouteUpdate guards in reused components.
  1. Call beforeEnter guards in the route config.
  1. Resolve async route components.
  1. Call beforeRouteEnter guards in activated components.
  1. Call global beforeResolve guards.
  1. Confirm navigation.
  1. Call global afterEach hooks.
  1. DOM updates triggered.
  1. Call callbacks passed to next in beforeRouteEnter guards with instantiated instances.
